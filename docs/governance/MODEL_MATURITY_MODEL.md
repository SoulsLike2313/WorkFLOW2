# MODEL MATURITY MODEL

Defines maturity levels for governance-driven repo operation.

## V1_STABLE

Definition:

- baseline controls exist and function.

Required capabilities:

1. repo-visible completion discipline
2. sync parity checks
3. basic governance stack present

Blocking weaknesses:

1. unresolved critical contradictions
2. unstable sync discipline

Promotion requirements:

1. repeated clean self-verification cycles
2. no critical blockers

## V1_PLUS

Definition:

- V1 with stronger consistency and reporting coverage.

Required capabilities:

1. deterministic read order in manifests
2. contradiction and drift checks integrated in routine workflow
3. machine-readable status artifacts

Blocking weaknesses:

1. missing governance references in manifests/docs
2. incomplete bootstrap sequence

Promotion requirements:

1. governance compliance at least `PARTIAL` without critical blockers
2. bundle/export readiness proven

## V1_EVOLVING

Definition:

- evidence-based evolution layer active, decisions no longer ad-hoc.

Required capabilities:

1. evolution signal registry operational
2. policy evolution log maintained
3. candidate-level recommendation generated from evidence

Blocking weaknesses:

1. weak evidence linkage
2. unresolved major drift trends

Promotion requirements:

1. stable control center full-check cycles
2. decreasing repeated failures

## V2_CANDIDATE

Definition:

- system demonstrates sustained control maturity and readiness for generation transition.

Required capabilities:

1. trust/sync/governance/admission all consistently green or justified conditional
2. critical contradiction prevention before escalation
3. hardening actions demonstrably reduce drift

Blocking weaknesses:

1. recurring hidden blockers
2. false PASS incidents

Promotion requirements:

1. readiness score threshold met
2. blocking signal set empty

## V2_READY

Definition:

- promotion criteria met; transition can be recommended with low risk.

Required capabilities:

1. sustained sync discipline
2. sustained self-verification reliability
3. policy evolution loop proven useful
4. operational usefulness of machine-proposed improvements demonstrated

Blocking weaknesses:

1. unresolved critical blocker
2. unstable evidence quality

Promotion requirements:

1. recommendation = `PROMOTE`
2. no active blocking signals

## V2_LOCKED

Definition:

- post-promotion level where controls are frozen except explicit evolution tasks.

Required capabilities:

1. strict change admission by hierarchy
2. regression prevention checks
3. stable policy baseline tracking

Blocking weaknesses:

1. uncontrolled policy changes
2. unresolved contradiction introduced after lock

Promotion requirements:

1. explicit governance mandate
2. successful lock audit
