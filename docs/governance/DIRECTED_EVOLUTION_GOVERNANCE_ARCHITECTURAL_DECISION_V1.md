# DIRECTED_EVOLUTION_GOVERNANCE_ARCHITECTURAL_DECISION_V1

Status:
- decision_state: `ACTIVE`
- decision_id: `DEVGOV-ADR-001`

## Decision

Adopt `Directed Evolution Governance V1` (clustered evolution model) as the mandatory policy for evolving repository zones.

Core model:
1. every evolving initiative must belong to a declared cluster,
2. every cluster has allowed vectors and forbidden drift vectors,
3. every candidate requires owner-visible rationale and admission gate before promotion.

## Why Now

Current repository maturity already includes:
1. governance core,
2. live owner observation layer,
3. product hardening execution lanes,
4. owner-character control layer.

At this point, the dominant risk is no longer missing foundations, but side evolution:
1. local optimization without shared direction,
2. visual/interactive branches growing faster than truth/hardening baseline,
3. hidden drift disguised as "innovation".

This is the canonical point where directional control yields highest benefit.

## Why Clustered Model (vs Free Evolution)

Clustered model is chosen because it:
1. preserves strategic coherence across multiple evolving zones,
2. forces bounded intent and measurable utility,
3. blocks swamp-side branches early,
4. keeps owner control explicit and reviewable.

Free evolution was rejected because it tends to:
1. blur priorities,
2. accumulate decorative complexity,
3. generate untraceable branch growth,
4. weaken evidence discipline.

## Rejected Alternatives

### A) Unclustered "let teams evolve naturally"

Rejected:
1. no bounded corridor,
2. no anti-noise admission barrier,
3. high drift probability under workload pressure.

### B) Single hard global lane only

Rejected:
1. over-constrains legitimate heterogeneous growth,
2. prevents meaningful owner-observation and product-surface evolution.

### C) Clustered model without owner gate

Rejected:
1. allows hidden promotion by local momentum,
2. breaks owner-controlled evolution principle.

## Hard Boundaries

1. policy cannot override repo law/governance hierarchy,
2. no hidden learning or hidden promotion,
3. no cross-cluster authority leaks without explicit boundary rule,
4. no completion/promotion claims without repo-visible evidence.
