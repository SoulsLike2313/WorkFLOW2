- status: `active`
- scope: `E:\CVVCODEX`

- default_for: `Codex execution + bundle/report workflow`

## Purpose

Set permanent token/limit economy mode that reduces repeated overhead without reducing truth, evidence, validation, or governance discipline.

## Hard Boundary

Limit economy mode never overrides:

1. source-of-truth precedence,
2. hard execution gates,
3. contradiction/gap disclosure,
4. validation evidence requirements,
5. one-main-truth-bundle law.

If economy preference conflicts with truth discipline, truth discipline wins.

## Permanent Default

Unless owner explicitly requests expanded mode, execution stays in economy mode:

1. stable canon knowledge is read from repo surfaces (not reprinted in prompt/chat),
2. chat stays short and delta-focused,
3. full detail goes to bundle/report artifacts,
4. each active step declares one primary truth bundle.

## Core + Delta Principle

Execution prompting and reporting follow:

1. `CORE`: stable law/architecture/governance surfaces already stored in repo,
2. `DELTA`: one active front, one primary bundle line, one exact remainder, one exact next move.

Full CORE reprint in every prompt is forbidden unless explicitly required by owner.

## Safe Compression Lower Bound

Lower bound is block-complete, not numeric-only:

1. compression may shorten wording,
2. compression may not remove mandatory control blocks (primary truth, scope, verdict, open risk, next step, owner decision trigger when required),
3. if a shorter format hides truth or owner-choice boundaries, it is invalid even in economy mode.

## Anti-Distortion Rule

Economy mode may compress repetition only.
It must never compress away:

1. source disclosure,
2. validation truth,
3. contradiction remainder,
4. remaining gaps,
5. active truth-binding state,
6. owner checkpoint posture honesty,
7. primary bundle declaration,
8. exact review paths.

## Operational Economy Rule

Routine startup work must not consume repeated Codex cycles when stable local runtime can be OS-managed.
For owner window and similar local services: prefer persistent localhost runtime with explicit logs and restart posture.

## Owner Override Window

Owner may request expanded response mode for a step.
Even with override, forbidden:

1. hiding unresolved gaps,
2. removing required evidence surfaces,
3. claiming completion without required bundle/validation chain.
