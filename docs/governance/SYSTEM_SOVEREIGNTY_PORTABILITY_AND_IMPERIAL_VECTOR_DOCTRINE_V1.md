# SYSTEM_SOVEREIGNTY_PORTABILITY_AND_IMPERIAL_VECTOR_DOCTRINE_V1

Status:
- class: `foundation_system_vector_doctrine`
- mutability: `immutable_by_default`
- change_authority: `EMPEROR_ONLY`
- validation_state: `doctrine_layer_active_not_full_runtime_implemented`

## Purpose

Fix long-term system vector without pretending full infra completion.

Covers:
1. repository sovereignty,
2. portable emperor machine bootstrap vector,
3. two-disk + N-backup storage doctrine,
4. disaster recovery to last canonical working step,
5. imperial multi-project dashboard vector.

## A) Repository Sovereignty

1. sovereign source of truth is local canonical repo only (`E:\CVVCODEX`),
2. safe mirror / external Git surfaces are non-sovereign derivatives,
3. external safe/export surface is manual and sovereign-decision-driven,
4. external surface is never default canonical path,
5. recoverability must be strengthened internally when external default is removed.

## B) Portable Emperor Machine Bootstrap Vector

Future bring-up on clean machine must follow deterministic layers:
1. entry layer (identity + foundation pointers),
2. dependency/install layer,
3. startup orchestration layer,
4. validation layer,
5. restore layer (to last canonical step).

Boundary:
- this doctrine is vector/foundation only, not claim of fully validated portable automation.

## C) Two-Disk + N-Backup Storage Doctrine

Base model:
1. `disk-1`: live system + primary runtime working surface,
2. `disk-2`: logs/history/backups/recovery support surface.

Scalable extension:
1. backup topology may scale to `N` backup disks,
2. read/write roles remain explicit and non-overlapping,
3. restore metadata must keep canonical step identity.

## D) Disaster Recovery To Last Canonical Step

Recovery target is not approximate date.
Recovery target is:
1. last canonical working step with evidence,
2. explicit failure point,
3. deterministic rebuild path to that step.

Future restore flow vector:
1. dead primary disk detected,
2. new blank primary prepared,
3. restore from backup/history to last canonical step,
4. continue in canon from restored state.

Boundary:
- no live physical migration tests are implied by this doctrine step.

## E) Imperial Dashboard Vector

Dashboard vector is system-wide, not single-project.

Target posture:
1. multi-project,
2. multi-node,
3. emperor-centered,
4. remote-ready,
5. role-filtered visibility,
6. honest truth-projection,
7. evolving with system maturity.

Aquarium vector:
1. human-readable live observation,
2. premium smooth visual layer,
3. no fake realtime claims.

Visibility boundary:
1. worker/limited nodes see bounded view,
2. Emperor view is full-scope.

## Non-Goals

1. no broad infra rewrite,
2. no fake claim of fully implemented node/server empire,
3. no replacement of existing self-description layer,
4. no default reintroduction of safe mirror as sovereign path.
