# CHEAPEST_VIABLE_FEDERATED_EXECUTION_MODEL_V1

Status:
- model_version: `v1`
- scope: `low-cost practical operating model for federated production`
- budget_assumption: `3-5 GPT/Codex subscriptions, one owner, several PCs`

Assertion labels:
- `PROVEN`
- `REUSABLE`
- `DESIGNED`
- `NOT YET IMPLEMENTED`

## 1) Cheapest realistic setup

Baseline:
1. one Emperor machine (law/signoff/escalation boundary);
2. one Primarch machine (department processor);
3. two Astartes machines (bounded execution).

This is enough to run first production pipeline without enterprise platform spend.

## 2) Why repo-first human-orchestrated federation now

1. cheaper than building full orchestration platform upfront;
2. easier to audit and correct;
3. lower lock-in risk;
4. easier to scale after evidence shows what truly matters.

## 3) What deterministic scripts/tools should do

1. validation and gate checks;
2. bundle packing/indexing;
3. queue state normalization;
4. report consistency checks.

## 4) What LLM should do

1. analysis synthesis;
2. risk reasoning;
3. roadmap and decomposition drafts;
4. bounded implementation support under explicit tasks.

## 5) Role cost discipline

### Emperor

Should spend time on:
1. sovereign decisions;
2. final escalations;
3. final product acceptance boundaries.

Should not spend time on:
1. routine decomposition;
2. low-level implementation tasks.

### Primarch

Should spend time on:
1. department routing;
2. synthesis quality;
3. workload balancing and escalations.

### Astartes

Should spend time on:
1. bounded execution;
2. reproducible evidence generation.

Should not spend time on:
1. strategy redesign;
2. scope negotiation politics.

## 6) Maximum output from limited resources

1. Keep one active department head first (Analytics).
2. Reuse fixed bundle templates and queue protocols.
3. Enforce entry manifests before heavy assignment.
4. Prioritize risk-first tasks over cosmetic tasks.
5. Batch similar tasks to reduce switching overhead.

## 7) What should NOT be built yet

1. full enterprise workflow platform;
2. autonomous planner with broad write authority;
3. multi-tenant cloud orchestration layer;
4. high-cost custom agent runtime before pilot evidence.

## 8) Current status

1. `PROVEN`: repo-first + governance-first discipline.
2. `DESIGNED`: cheapest viable production operation profile.
3. `NOT YET IMPLEMENTED`: full automation around this profile.
