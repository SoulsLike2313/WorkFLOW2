# EXPLAINABILITY_OPTIMIZATION_SUMMARY

## Scope
Governance/policy/machine-brain/repo-control explainability layer only.
No product-project implementation changes.

## 1) What was improved
- Added human-readable governance explainability documents.
- Added one-screen machine status model with compact JSON snapshot.
- Added plain-language runtime status page generated by Repo Control Center.
- Split status semantics into independent layers: workspace, authority, governance, admission.

## 2) New human-readable artifacts
- `docs/governance/MACHINE_OPERATOR_GUIDE.md`
- `docs/governance/MACHINE_CAPABILITIES_SUMMARY.md`
- `docs/governance/POLICY_DIGEST.md`
- `runtime/repo_control_center/plain_status.md`
- `runtime/repo_control_center/one_screen_status.json`

## 3) What is now faster to understand
- Current machine mode and creator authority state.
- Whether trust/sync/governance/admission are green.
- Exact blocker category and detail.
- Next canonical step without reading the full policy stack.

## 4) Status-layer separation result
Yes. The status model now separates:
- `workspace_health`
- `authority_status`
- `governance_status`
- `admission_status`

## 5) One-screen status generation
Yes.
- Generated by `scripts/repo_control_center.py`
- Output: `runtime/repo_control_center/one_screen_status.json`

## 6) Plain-language summary generation
Yes.
- Generated by `scripts/repo_control_center.py`
- Output: `runtime/repo_control_center/plain_status.md`

## 7) Remaining explainability gaps
- Evolution wording can still be simplified for non-technical operators.
- Policy digest may need periodic refresh when governance files are added/retired.

## 8) Next explainability step
- Keep one-screen status as primary operator surface.
- Keep policy digest synchronized with governance hierarchy changes.
- Keep blocker categories stable (`SYNC`, `ROLE_AUTHORITY`, `GOVERNANCE_POLICY`, `ADMISSION_GATE`, `TRUST`).
